1. SYSTEM OVERVIEW
================================================================================

The Challenge System is a comprehensive platform for creating and managing 
ethical hacking learning experiences. It follows a client-server architecture 
with FastAPI backend and HTML/JavaScript frontend, utilizing Docker containers 
for isolated challenge environments.

Key Components:
- Admin Panel: Challenge creation and management
- User Portal: Challenge discovery and completion
- Docker Integration: Containerized challenge environments
- Database: SQLite3 with Foreign Key constraints
- API Layer: RESTful endpoints for all operations

2. DATABASE SCHEMA & STRUCTURE
================================================================================

2.1 CHALLENGES TABLE
--------------------
CREATE TABLE challenges (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    category TEXT,              # e.g., 'Web', 'Network', 'Cryptography'
    difficulty TEXT,            # 'Easy', 'Medium', 'Hard'
    points INTEGER,             # XP points awarded for completion
    description TEXT,           # Challenge description/instructions
    flag TEXT,                  # Secret flag users must find
    docker_image TEXT NOT NULL, # Docker image to run for this challenge
    is_published INTEGER DEFAULT 0,  # 0=Draft, 1=Published
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)

2.2 USER_CHALLENGES TABLE (User-Challenge Relationship)
---------
CREATE TABLE user_challenges (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER,
    challenge_id INTEGER,
    solved_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    progress INTEGER DEFAULT 0,    # 0-100% completion
    UNIQUE(user_id, challenge_id), # One entry per user-challenge pair
    FOREIGN KEY(user_id) REFERENCES users(id),
    FOREIGN KEY(challenge_id) REFERENCES challenges(id)
)

Purpose: Tracks which challenges each user has solved and their progress

2.3 RUNNING_CONTAINERS TABLE (Active Sessions)
----------
CREATE TABLE running_containers (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    challenge_id INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    container_id TEXT NOT NULL,     # Docker container ID
    container_name TEXT,
    port_mapping TEXT,              # Port exposed to user
    status TEXT DEFAULT 'running',  # Status: 'running'/'stopped'
    started_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    stopped_at TIMESTAMP,
    duration_minutes INTEGER DEFAULT 30,
    FOREIGN KEY(user_id) REFERENCES users(id),
    FOREIGN KEY(challenge_id) REFERENCES challenges(id)
)

Purpose: Tracks active Docker containers for each user-challenge session

2.4 CHALLENGE_SOLVES TABLE (Statistics)
----------
CREATE TABLE challenge_solves (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    challenge_id INTEGER UNIQUE NOT NULL,
    title TEXT NOT NULL,
    category TEXT,
    count INTEGER DEFAULT 0,        # Number of users who solved it
    FOREIGN KEY(challenge_id) REFERENCES challenges(id) ON DELETE CASCADE
)

Purpose: Tracks completion statistics for analytics

3.1 CHALLENGE CREATION PROCESS
-------------------------------

Step 1: Admin navigates to Admin Panel → Challenges
        URL: /admin/challenges

Step 2: Admin clicks "Create Challenge" button
        - Modal opens: create-modal-content
        - UI Components:
          * Title input field
          * Category dropdown (Web, Network, Cryptography, etc.)
          * Difficulty dropdown (Easy, Medium, Hard)
          * Points input (XP value for completion)
          * Description textarea (with markdown support)
          * Flag input (secret answer users must find)
          * Docker Image selector (with preview modal)

Step 3: Admin fills challenge details
        Example:
        {
            "title": "SQL Injection Attack",
            "category": "Web",
            "difficulty": "Medium",
            "points": 250,
            "description": "Exploit a vulnerable login page using SQL injection...",
            "flag": "FLAG{sql_admin_password}",
            "docker_image": "sql-injection-challenge:latest"
        }

Step 4: Admin clicks "Create" button
        - Form data sent via POST to /admin/challenges/create
        - Validation occurs (all required fields checked)
        - Challenge inserted into database with is_published=0 (Draft)
        - Success message shows: "Challenge saved successfully"
        - Challenge ID returned to admin

3.2 CHALLENGE CREATION ENDPOINT
-------------------------------

POST /admin/challenges/create

Request Body:
{
    "title": string (required),
    "category": string,
    "difficulty": string,
    "points": integer,
    "description": string (required),
    "flag": string (required),
    "docker_image": string (required)
}

Validation:
- Title, Description, Flag are required and non-empty
- Docker image is mandatory (challenge needs container)
- All fields are trimmed of whitespace

Response (Success):
{
    "success": true,
    "id": 5,
    "message": "Challenge saved successfully"
}

Response (Failure):
{
    "success": false,
    "error": "Missing required fields"
}

Database Effect:
INSERT INTO challenges (title, category, difficulty, points, 
description, flag, docker_image, is_published)
VALUES (..., ..., ..., ..., ..., ..., ..., 0)

4. CHALLENGE PUBLICATION & MANAGEMENT
================================================================================

4.1 PUBLISH CHALLENGE (Make Available to Users)
-----------------------------------------------

POST /admin/challenges/{challenge_id}/publish

Purpose: Change is_published from 0 to 1

Before Publishing:
- Challenge is only visible to admins
- Users cannot see or attempt it
- Status shows as "Draft"

After Publishing:
- Challenge becomes available to all users
- Appears in user challenge list
- Users can start attempting if unlocked

Database Update:
UPDATE challenges SET is_published=1 WHERE id=?

4.2 UNPUBLISH CHALLENGE
-----------------------

POST /admin/challenges/{challenge_id}/unpublish

Purpose: Hide challenge from users (is_published: 1 → 0)
- Remove from user view
- Stop new user attempts
- Existing attempts not affected

4.3 DELETE CHALLENGE
--------------------

POST /admin/challenges/{challenge_id}/delete

Purpose: Permanently remove challenge
- Deletes from challenges table
- Cascades to user_challenges (foreign key)
- Stops all running containers
- Removes from challenge_solves statistics

Cascade Cleanup:
- user_challenges records deleted
- challenge_solves records deleted
- running_containers stopped

5.2 CHALLENGE LOCKING MECHANISM
------------------------------

Progressive Unlocking:
- First challenge (by creation order) is always UNLOCKED
- Subsequent challenges are LOCKED until previous is completed
- Users must solve challenges sequentially

Lock Status Logic:
if challenge_position > 0:
    previous_challenge_id = all_challenges[challenge_position - 1]
    if previous_challenge_id NOT IN completed_challenges:
        is_locked = true  # Cannot attempt until unlocked

Benefits:
✓ Ensures learning progression
✓ Prevents users from skipping foundational concepts
✓ Gamification element (unlock as you succeed)
✓ Ordered difficulty progression

6. CHALLENGE ATTEMPT WORKFLOW
================================================================================

6.1 STARTING A CHALLENGE
-----------------------

User clicks "Start Challenge" button on challenge card

Step 1: Request validation
        - Check if user exists
        - Check if challenge exists
        - Check if challenge is published
        - Check if challenge is unlocked

Step 2: Container Start
        POST /user/challenges/{challenge_id}/start-container?email=user@example.com
        
        Process:
        a) Fetch challenge Docker image from database
        b) Call DockerManager.start_container()
        c) Docker container spins up on random available port
        d) Container health checked
        e) Running container info stored in database

Step 3: Container Registration
        INSERT INTO running_containers (
            challenge_id, user_id, container_id, 
            container_name, port_mapping, status='running'
        ) VALUES (...)

Step 4: User Redirected
        - Redirect to challenge attempt page
        - Container port/URL provided to user
        - User can now interact with vulnerable application in container

Cleanup of Old Containers:
If user already has running container for same challenge:
- Stop previous container
- Delete old record
- Start fresh container (prevents conflicts)

6.3 FLAG SUBMISSION & VERIFICATION
---------------------------------

User finds flag in the vulnerable application or environment

POST /user/api/challenges/{challenge_id}/verify-flag

Request:
{
    "email": "user@example.com",
    "flag": "FLAG{sql_admin_password}"
}

Verification Process:
1. Fetch challenge from database
2. Retrieve stored flag: SELECT flag FROM challenges
3. Compare: submitted_flag == stored_flag
4. String trimmed and compared

Response (Correct):
{
    "success": true,
    "message": "Challenge solved! +250 XP",
    "points_earned": 250,
    "user_level": 5,
    "user_xp": 1250
}

Response (Incorrect):
{
    "success": false,
    "message": "Flag is incorrect. Try again!",
    "points_earned": 0
}

On Correct Flag:
1. Mark challenge as completed: progress=100
2. Award XP to user: user.xp += challenge.points
3. Increment challenges_solved counter
4. Record solve time
5. Update ranking table
6. Unlock next challenge

6.4 STOPPING A CHALLENGE
-----------------------

User clicks "Stop Challenge" or time expires

POST /user/challenges/{challenge_id}/stop-container?email=user@example.com

Process:
1. Get running container ID for user-challenge
2. Call DockerManager.stop_container(container_id)
3. Container stops and cleans up resources
4. Update database: status='stopped', stopped_at=NOW

Database Update:
UPDATE running_containers 
SET status='stopped', stopped_at=CURRENT_TIMESTAMP 
WHERE container_id=?

Resources Freed:
✓ Container stopped
✓ Port released for reuse
✓ Memory freed
✓ Network connections closed
✓ Database record marked as stopped
