================================================================================
                    DOCKER CONTAINER MANAGEMENT
================================================================================

FILE: backend/docker_manager.py (~300+ lines)

PURPOSE: Orchestrate Docker containers for challenge applications
LANGUAGE: Python 3.9+
KEY FEATURE: Handles starting, stopping, monitoring challenge containers

CRITICAL FUNCTIONALITY:

1. PORT ALLOCATION
   - Uses port range: 16000-18000 (shifted high to avoid TIME_WAIT conflicts)
   - Allocates TWO consecutive ports per challenge
   - Checks port availability using socket binding
   - Implements port tracking cache: _container_ports dict
   - Exponential backoff for port reservation

2. CONTAINER LIFECYCLE MANAGEMENT
   start_container(docker_image, challenge_id, user_id, is_preview=False)
   - Initializes Docker client
   - Gets available port pair
   - Cleans up existing containers (docker rm -f)
   - Special cleanup for preview containers
   - Determines exposed ports from image metadata
   - Builds port mapping (5000/tcp → host_port, 8000/tcp → host_port_alt)
   - Runs container via CLI: docker run -d -p 127.0.0.1:PORT:CONTAINERPORT
   - Maps ONLY to 127.0.0.1 (localhost) for security
   - Health check: waits up to 15 seconds for port to respond
   - Returns: {"success": true, "container_id": id, "container_name": name, "port": port}

3. DOCKER CLIENT MANAGEMENT
   _get_docker_client()
   - Lazy initialization pattern
   - Connects to Docker daemon via socket
   - Fallback handling if daemon unavailable
   
   _initialize_client()
   - Attempts client.from_env()
   - Calls client.ping() to verify
   - Graceful failure if daemon not running

4. HEALTH CHECKS
   - Socket-based connectivity check
   - Timeout: 0.5 seconds per attempt
   - Retries: 15 attempts (7.5 seconds total)
   - Verifies container is responsive before returning success

5. ERROR HANDLING
   - Catches OperationalError for lock handling
   - Returns structured error responses
   - Logs all major operations to stdout

1. BRUTE-FORCE CHALLENGE
   Location: challenges/brute-force/
   Port: 5000 (via Docker mapping)
   
   Vulnerability: No rate limiting, no lockout mechanism
   Files:
   - app.py: Flask application
   - Dockerfile: Python 3.9-slim base, port 5000
   - requirements.txt: flask
   - templates/login.html: Login form
   - templates/dashboard.html: Success page
   
   Target Credentials:
   - Username: "admin"
   - Password: "111"
   
   Endpoints:
   - GET /: Login form
   - POST /: Submit username/password
   - GET /dashboard: Success page (after login)
   - GET /logout: Clear session

2. COMMAND-INJECTION CHALLENGE
   Location: challenges/command-injection/
   Port: 5000
   
   Vulnerability: shell=True with direct user input in subprocess
   Flaw: subprocess.check_output(f"ping -c 4 {ip_address}", shell=True)
   
   Endpoints:
   - GET /: Form requesting host IP
   - POST /: Executes "ping -c 4 {user_input}" via shell
   - GET /flag: Hidden flag endpoint returns FLAG{C0MM4ND_INJ3CT10N_W1Z4RD_2026}
   
   Attack Vector: Inject shell commands via IP input field

3. CSRF-CHALLENGE (Cross-Site Request Forgery)
   Location: challenges/csrf-challenge/
   Port: 5000
   
   Vulnerability: No CSRF protection, session-based auth without tokens
   
   Initial Balances:
   - admin: $1000
   - victim: $5000
   
   Endpoints:
   - GET /: Dashboard showing user balance and transactions
   - POST /transfer: Vulnerable transfer endpoint (no CSRF token)
     * Parameters: to (recipient), amount (integer)
     * Logic: deducts from sender, adds to recipient
     * History: logged in transactions list
   - GET /flag: Returns flag if admin balance > $2000
     * Flag: FLAG{CSRF_B4NK_HACK_2026}
     * Need to steal $1000+ from victim to admin via CSRF
   - GET /attacker: Attacker page (likely contains CSRF form)
   - GET /reset: Resets balances and transactions
   - GET /logout: Clear session

4. FILE-UPLOAD-VULNERABILITY
   Location: challenges/file-upload-vulnerability/
   Port: 5009
   
   Vulnerability: No file type/extension validation, arbitrary code execution
   
   Endpoints:
   - GET /: File upload form
   - POST /upload: Uploads file to /uploads directory
     * No MIME type checking
     * Accepts any extension
     * If .py file: executes it via subprocess.run(['python', file_path])
     * Returns output to user
   - GET /uploads/<filename>: Download uploaded file
   - Creates: /uploads/flag.txt with FLAG{INSECURE_F1L3_UPLOAD_2026}

5. IDOR-CHALLENGE (Insecure Direct Object Reference)
   Location: challenges/idor-challenge/
   Port: 5004
   
   Vulnerability: Direct object reference to admin user without authorization
   
   Mock Users Database:
   - User ID "9": alice_w (UX Designer) - Regular user
   - User ID "2": bob_s (Systems Analyst) - Regular user
   - User ID "30": admin_master (System Administrator) - ADMIN
   - User ID "4": charlie_d (DevOps Engineer) - Regular user
   
   Each user has:
   - username, full_name, role, email, bio
   - joined, last_login, privacy_token
   - is_admin flag
   
   Admin User (ID 30):
   - privacy_token: FLAG{IDOR_SYSTEM_BR3ACH_2026}

   - GET /: Redirects to /profile/9 (logged in as alice_w)

6. LFI-CHALLENGE (Local File Inclusion)
   Location: challenges/lfi-challenge/
   Port: 5006
   
   Vulnerability: Path traversal in file reading
   
   Endpoints:
   - GET /?page=<filename>: Reads file from pages directory or absolute path
